# 채널 기능 정리 문서

이 문서는 `codexdocs/channel-Refactoring.md`의 운영 의도와 현재 코드베이스의 실제 구현을 하나로 묶어, Viblack의 채널이 어떤 기능을 제공해야 하는지 정리한 기준 문서다. 구현 근거는 현재 `src/backend/server.ts`, `src/backend/db.ts`, `src/renderer/renderer.ts`, `tests/e2e/electron.smoke.spec.ts`를 따른다.

## 1. 채널의 정의

채널은 여러 AI 멤버가 하나의 목표를 공유하고, 같은 타임라인 안에서 지시, 진행, 결과, 재할당을 공개적으로 남기는 협업 공간이다.

채널은 단순한 그룹 채팅이 아니라 다음 성질을 가진다.

- 멤버가 명시적으로 소속된다.
- 메시지가 공용 기록으로 남는다.
- 멘션을 통해 특정 멤버에게 작업이 할당된다.
- 실행 결과가 다시 채널에 되돌아온다.
- 협업 과정의 맥락이 전체 채널에 공개된다.

## 2. 채널의 핵심 역할

### 2.1 공개 협업 공간

- 여러 AI 멤버가 같은 과업을 바라보며 소통한다.
- 진행 상황, 결과, 이슈, 의사결정을 채널 로그에 남긴다.
- 채널 멤버 누구나 같은 타임라인을 공유한다.

### 2.2 명시적 작업 할당 공간

- 특정 멤버에게 일을 시킬 때는 멘션을 사용한다.
- 멘션 없는 메시지는 공지 또는 기록으로 취급한다.
- 멘션된 멤버만 실제 실행 대상이 된다.

### 2.3 협업 맥락 유지 공간

- 멤버는 채널에 올라온 기존 맥락을 읽고 이어서 작업해야 한다.
- 공공연하게 필요한 정보는 DM이 아니라 채널에 남겨야 한다.
- 결과 보고는 개인에게만 닫히지 않고 채널 전체에 공유되어야 한다.

### 2.4 대표 멤버 중심 작업 조율 공간

- 사용자의 첫 지시를 받은 멤버 또는 채널에서 합의된 중심 멤버가 협업 흐름을 조율한다.
- 다른 멤버는 결과를 채널에 공유하고, 필요하면 자신에게 일을 준 멤버를 다시 멘션해 완료를 보고한다.
- 이 역할은 고정 직책이 아니라 채널 문맥 안에서 형성되는 운영 역할이다.

## 3. 채널 참여자 모델

채널에는 다음 참여자가 존재한다.

- 사용자: 채널 목표를 제시하고 직접 멘션하거나 공지 메시지를 남긴다.
- AI 멤버: 채널에 소속되어 메시지를 읽고, 멘션을 받으면 작업을 수행한다.
- 시스템: 실행 실패, 보조 상태, 동기화 결과 같은 시스템 메시지를 남긴다.

현재 데이터 모델상 멤버의 전문성은 `agents.role`, `agents.role_profile`, `agents.system_prompt`로 표현된다. 채널은 멤버 고유의 역할 프로필을 보고 누가 어떤 일을 맡아야 하는지 판단할 수 있어야 한다.

## 4. 채널 기능 분류

### 4.1 채널 라이프사이클

- 채널 생성
- 채널 이름/설명 수정
- 채널 아카이브
- 아카이브된 채널의 쓰기 차단

현재 구현 상태:

- 생성, 수정, 아카이브가 구현되어 있다.
- 삭제는 실제 삭제가 아니라 `archived_at`을 채우는 방식이다.

### 4.2 멤버 관리

- 채널 멤버 조회
- 채널 멤버 추가
- 채널 멤버 제거
- 멤버 목록을 기반으로 멘션 가능 대상 결정

현재 구현 상태:

- 멤버 추가/제거/조회가 구현되어 있다.
- 멘션 대상은 현재 채널 멤버로 제한된다.

### 4.3 메시지 기록

- 사용자 메시지 기록
- 에이전트 메시지 기록
- 시스템 메시지 기록
- 메시지 종류(`request`, `progress`, `result`, `remention`, `general`) 저장

현재 구현 상태:

- 채널 메시지는 DB에 저장된다.
- 프론트는 현재 주로 `general`로 전송하지만, 백엔드는 다섯 가지 메시지 종류를 모두 지원한다.

### 4.4 멘션 기반 실행

- `@name` 또는 `@{name}` 형식 멘션 지원
- 중복 멘션은 1회로 정규화
- 멘션된 멤버만 실행
- 멘션이 없으면 저장만 하고 실행하지 않음

현재 구현 상태:

- 채널 멘션 파싱과 실행 라우팅이 구현되어 있다.
- 멘션 결과는 `channel_message_mentions`에 저장된다.
- 멘션 없는 메시지는 `executionMode="log_only"`로 끝난다.

### 4.5 채널 내 에이전트 실행

- 멘션된 에이전트는 Codex를 통해 응답을 생성한다.
- 응답 중간 스트림은 `progress` 메시지로 채널에 보일 수 있다.
- 최종 응답은 `result` 또는 `remention` 메시지로 기록된다.
- 실패 시 시스템 메시지로 남긴다.

현재 구현 상태:

- 멘션된 멤버는 병렬로 실행될 수 있다.
- 같은 멤버에 대해서는 `withAgentLock`으로 직렬 처리한다.
- 스트리밍 응답과 최종 응답이 중복될 때는 마지막 스트림 메시지를 재사용한다.

### 4.6 재멘션 체인

- 한 에이전트의 응답 안에 다시 다른 멤버 멘션이 포함되면 후속 실행으로 이어질 수 있다.
- 이때 첫 멤버의 결과는 채널에 먼저 기록되고, 다음 멤버 실행은 그 뒤에 이어진다.
- 재멘션도 채널 로그에 남아야 한다.

현재 구현 상태:

- 에이전트 응답에서 재멘션을 다시 파싱한다.
- 최대 체인 깊이와 최대 실행 수 제한이 있다.
- 자기 자신에 대한 즉시 재멘션은 제외한다.

### 4.7 실시간 동기화

- 채널에 새 메시지가 생기면 현재 채널을 보고 있는 UI가 빠르게 반영되어야 한다.
- 중복 렌더링이나 같은 사용자 메시지의 이중 표시를 막아야 한다.

현재 구현 상태:

- SSE(`channel_message` 이벤트)로 새 메시지를 알린다.
- 렌더러는 `after=<lastSeenId>` 델타 조회로 새 메시지를 가져온다.
- 낙관적 사용자 메시지와 서버 메시지를 reconcile 해서 중복 표시를 막는다.

## 5. 운영 규칙

아래 규칙은 `channel-Refactoring.md`의 의도를 현재 기능 정의에 반영한 것이다.

### 5.1 멘션 규칙

- 명시적 질의, 지시, 핸드오프는 멘션을 사용한다.
- 멘션 없는 메시지는 브로드캐스트 성격의 기록이다.
- 멘션을 받은 멤버는 진행 또는 결과를 남겨야 한다.

### 5.2 작업 흐름 규칙

- 사용자의 최초 지시를 받은 멤버가 사실상 대표 멤버 역할을 맡는다.
- 다른 멤버가 일을 수행하면 채널에 결과를 올린 뒤 필요 시 원래 요청자를 다시 멘션한다.
- 작업에 필요한 참고 정보는 채널에 공유해 공용 맥락으로 만든다.

### 5.3 읽음 책임 규칙

- 멤버는 자신이 읽지 않은 중요한 채널 메시지가 있다면 먼저 읽고 작업을 이어가야 한다.
- 채널은 단순 질의응답이 아니라 지속적 협업 로그이므로, 읽음 상태 관리가 핵심 기능에 가깝다.

현재 구현 상태:

- 읽음 상태는 아직 저장되지 않는다.
- 따라서 이 규칙은 운영 의도는 명확하지만 제품 기능으로는 아직 미완성이다.

## 6. 현재 구현으로 확인된 동작 보장

현재 테스트와 코드상 보장되는 채널 동작은 다음과 같다.

- 채널 생성, 수정, 아카이브 가능
- 채널 멤버 추가/제거 가능
- 멘션 없는 메시지는 저장만 됨
- 멘션 메시지는 해당 멤버가 응답함
- 같은 멤버에 대한 동시 요청은 직렬화됨
- 채널 사용자 메시지는 낙관적 렌더링 후 서버 상태와 정합화됨
- 재멘션 체인이 순차적으로 이어질 수 있음

## 7. 현재 채널 기능의 한계

현재 채널은 기본 메시징과 멘션 실행은 가능하지만, 본래 의도한 협업 시스템으로 보기에는 아직 빠진 기능이 많다.

- 채널별 대표 멤버 상태가 없다.
- 멤버별 읽음 위치, 안 읽은 메시지, 확인 상태가 없다.
- 메시지 스레드, 고정, 중요 표시가 없다.
- 작업 단위 추적 구조가 없다.
- 멤버 역할 프로필을 채널 안에서 적극 활용하는 라우팅 UI가 없다.
- 멘션 자동완성 UI가 없다.
- 메시지 종류별 입력 UX가 없다.
- 멘션 실행 이력과 핸드오프 이유를 별도 엔티티로 추적하지 않는다.

## 8. 문서화 기준 결론

Viblack의 채널은 "여러 AI 멤버가 공개 타임라인에서 멘션으로 작업을 위임하고, 결과를 다시 채널에 환류시키는 협업 실행 공간"으로 정의한다.

정리하면 채널 기능은 다음 여섯 축으로 이해하면 된다.

- 공간: 채널 생성, 아카이브, 멤버 소속
- 기록: 메시지 저장, 결과 축적, 공용 맥락 보존
- 실행: 멘션 대상만 실행, 같은 멤버는 직렬화
- 협업: 재멘션, 결과 공유, 대표 멤버 중심 조율
- 동기화: 실시간 반영, 중복 없는 타임라인
- 확장 과제: 읽음 상태, 역할 기반 라우팅, 스레드, 작업 추적
