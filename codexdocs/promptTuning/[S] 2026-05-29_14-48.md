- 멤버 구성: 영희(시스템 기획자/coordinator), 존(리서치 전문가), 매튜(문서 작성 전문가)

## 2. 가설 및 기대 결과

가설:
- 채널 멤버가 요청을 받았을 때 자기 역할과 산출물의 적합성을 먼저 판단하도록 하면, 자기 전문 영역을 유지하면서 더 적합한 멤버에게 실행을 위임할 가능성이 높아진다.
- 이 규칙은 특정 역할별 금지 목록이 아니라 일반적인 역할-산출물 매칭 규칙이어야 한다.

기대 결과:
- coordinator는 자신의 역할에 맞게 범위, 요구사항, 수용 기준, 순서를 정리한다.
- 실제 조사/문서화/구현 같은 전문 실행은 CHANNEL_MEMBERS 중 적합한 멤버에게 위임한다.
- 역할 침범을 줄이되, 불필요한 중간 파일이나 질문 루프는 늘리지 않는다.

근거:

- 다만 `no_question_loop` 기준이 실패했다. 리포트는 `message 3에서 질문 후 사용자 응답 없이 후속 에이전트 실행이 이어졌다`고 판단했다.
- 기존 채널 규칙은 `delegate` 사용 조건과 coordinator/worker 종료 방식은 설명했지만, 요청 산출물이 자기 역할과 맞는지 판단하고 적합 멤버에게 넘기는 일반 규칙은 부족했다.

## 4. 시스템 프롬프트 수정 내역

수정 파일:
- `src/backend/prompt-templates/member-execution-channel-rules.md`

문제 행동:
- 역할이 다른 멤버가 같은 채널에 있어도, 요청을 받은 멤버가 자기 역할 산출물과 전문 실행 산출물을 구분해야 한다는 지시가 약했다.
- 1차 수정에서는 "기획자는 개발하지 말라"와 같은 역할별 금지문으로 흐를 위험이 있었다.

원인:
- 기존 프롬프트는 위임 action 형식 중심이었고, 역할 적합성 판단 기준이 명시되어 있지 않았다.

부족했던 지시:
- `CHANNEL_MEMBERS`의 역할을 읽고, 요청 산출물을 역할별 책임으로 나누는 규칙
- coordinator가 직접 전문 실행을 대체하는 대신 자기 역할에 맞는 범위/요구사항/수용 기준을 제공하는 규칙
- worker가 다른 전문 영역을 받았을 때 coordinator에게 역할 적합성 보고를 돌려주는 규칙

최종 수정 방향:
- 특정 역할별 금지 목록을 넣지 않는다.
- "기획자는 코딩하지 말라", "개발자는 그림 그리지 말라" 같은 엣지케이스 나열 대신, 역할-산출물 매칭 규칙으로 일반화한다.
- coordinator handoff는 본문 브리프로 유지하고, 명시 요청이 없는 중간 handoff 파일 생성을 줄인다.

핵심 추가 문장:
- `Before specialist work, map the requested deliverable to your own role and the roles in CHANNEL_MEMBERS. Own the parts that match your role, and route the remaining specialist execution to the best-suited member.`
- `If you are the coordinator and the request spans multiple specialties, contribute brief in-message scoping, requirements, constraints, acceptance criteria, or sequencing, then use type=delegate to the best-suited exact member for the next execution step. Create handoff files only when the user explicitly requested them.`
- `If you are a worker and the assignment belongs to another specialty, publish a concise role-fit report with the recommended owner, then use type=report back to the requester/coordinator.`
