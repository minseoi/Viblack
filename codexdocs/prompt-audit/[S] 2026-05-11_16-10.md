## 2. 가설 및 기대 결과

가설:

- 기존 프롬프트는 specialist 간 충돌을 최종안에 반영하도록 유도하지만, 남은 blocker마다 담당 owner/gate를 붙이라고 명시하지 않는다.
- 그 결과 최종안이 충분히 안전하더라도 실제 실행 인수인계에서는 “누가 무엇을 닫아야 하는지”가 약해질 수 있다.
- coordinator final 규칙에 owner/gate 요구를 추가하면 실행 가능성과 정보 보존율이 올라간다.

기대 결과:

- 최종 보고가 단순 리스크 나열에서 blocker별 담당/검증 gate 매핑으로 강화된다.
- 협업 단계 수, 역할 분리, handoff field coverage는 유지된다.
- action block protocol-only 규칙은 유지된다.

## 4. 시스템 프롬프트 수정 내역

문제: 최종 blocker의 담당 owner/gate가 불명확함

1. 어떤 행동이 문제였는가
   - coordinator가 충돌을 잘 요약했지만, 남은 blocker를 누가 닫아야 하는지 명확히 배정하지 않았다.
   - 최종안이 실행 계획처럼 보이지만 실제 handoff 관점에서는 담당 경계가 약하다.

2. 기존 프롬프트의 어떤 문구가 원인이었는가
   - `Every delegate/report/final handoff must preserve five compact fields...`
   - `Coordinators must not start a dependent phase until the prior worker's public report is present...`
   - 위 문구들은 정보 보존과 순차 진행은 강제하지만, 최종 충돌 조정 결과에 owner/gate를 붙이라는 요구는 없다.

3. 어떤 지시가 부족했는가
   - conflicting specialist reports를 final에서 조정할 때 accepted trade-off와 remaining blocker별 owner/gate를 명시하라는 지시.

4. 어떤 문장으로 수정했는가
   - `When the final answer reconciles conflicting specialist reports, list the accepted trade-offs and assign an owner or gate for each remaining blocker instead of only listing risks.`

5. 수정 후 기대 행동은 무엇인가
   - coordinator final이 단순 리스크 목록이 아니라 `무엇을 수용했는지`, `무엇이 남았는지`, `누가 닫아야 하는지`를 함께 전달한다.
