## 2. 가설 및 기대 결과

가설:

- 이전 프롬프트는 handoff 필드 보존은 강제하지만, 그 필드를 `CHANNEL_ACTION` 블록 안에 넣지 말라는 제한이 없다.
- 이 때문에 멤버가 `type`, `target`, `artifact_path` 외의 임의 key를 action block에 넣고, parser가 무시하는 정보를 프로토콜 필드처럼 보이게 만들 수 있다.
- 이는 시스템 로직 변경 없이 프롬프트 문구만으로 줄일 수 있다.

기대 결과:

- `CHANNEL_ACTION` block은 protocol-only 형식으로 유지.
- 결정 사항, 미해결 문제, 리스크, 의존성, 검증 필요 항목은 공개 본문에 보존.
- 멤버별 전문 역할, 협업 단계, 최종 실행안 품질은 유지.

## 4. 시스템 프롬프트 수정 내역

문제: action block protocol 오염

1. 어떤 행동이 문제였는가
   - 전문가들이 handoff 필드를 공개 본문에 보존하면서도 `CHANNEL_ACTION` block 내부에 임의 key로 중복 기입했다.
   - 예: `결정 사항=...`, `미해결 문제=...`, `리스크=...`가 action block에 들어감.

2. 기존 프롬프트의 어떤 문구가 원인이었는가
   - `Every delegate/report/final handoff must preserve five compact fields: 결정 사항, 미해결 문제, 리스크, 의존성, 검증 필요 항목.`
   - 이 문구는 필수 정보 보존 위치를 공개 본문으로 제한하지 않았다.

3. 어떤 지시가 부족했는가
   - `CHANNEL_ACTION` block에 허용되는 key가 `type`, `target`, `artifact_path`뿐이라는 명시.
   - handoff 필드는 public reply body에만 쓰라는 경계 지시.

4. 어떤 문장으로 수정했는가
   - `Keep the CHANNEL_ACTION block protocol-only. Put handoff fields in the public reply body, not inside CHANNEL_ACTION. The action block may contain only type, target, and artifact_path lines.`

5. 수정 후 기대 행동은 무엇인가
   - 공개 본문에는 handoff 필수 5개 항목이 남는다.
   - action block은 자동 실행 protocol에 필요한 최소 key만 포함한다.
   - parser가 무시하는 정보를 실행 계약처럼 착각하는 일이 줄어든다.
